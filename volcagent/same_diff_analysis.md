## 5. 关键发现

### 5.1 same.log 在测试初期集中触发冷启动链式长尾
- 前 30 个请求里连续出现 **3 次 >1 s 的延迟**（#5 1057 ms、#27 1286 ms、#28 1606 ms），且 #27、#28 是**同一秒内相邻**的两个请求。
- 这些 pod 在 same.log 整个测试期间**仅被命中 10~17 次**，每次首次/前几次命中都伴随明显高延迟（典型冷启动曲线）。
- 推测：same 模式把流量集中路由到一组**新建 pod**，多个冷启动请求几乎同时触发，导致连续出现秒级尖刺。

### 5.2 diff.log 长尾收敛更快
- diff 没有 >1 s 的请求，最大尖刺 782 ms。
- 两个最大延迟请求（#59、#90）都是**首次命中某个老 pod**，但延迟被限制在 800 ms 以内，说明**预热过的 pod 重新启用时延迟可控**。
- diff 模式下新 pod 数量较少（5 个 vs same 的 7 个新 pod），且流量被分散到已经预热的老 pod 上。

### 5.3 稳态下两者接近
去除 >500 ms 离群值后的"稳态平均"：
- same.log：(160.5 × 100 − 1606 − 1286 − 1057 − 642) / 96 ≈ **119 ms**
- diff.log：(131.3 × 100 − 782 − 746) / 98 ≈ **124 ms**

稳态下差异缩小到约 5 ms（**same 略优**，对应 P50 111 vs 114）。

### 5.4 P95 差距最显著
- same P95 = **223 ms**（被 #5/#27/#28 三个 1 s+ 离群直接拉上去）
- diff P95 = **143 ms**（与 P90 几乎重合，离群影响很小）

## 6. 结论与建议

| 维度 | same | diff | 评估 |
|---|---|---|---|
| 稳态延迟（P50/稳态均值） | 111 ms / 119 ms | 114 ms / 124 ms | 接近，same 略优 |
| P90 | 152 ms | 135 ms | diff 更优 |
| **P95** | **223 ms** | **143 ms** | **diff 显著更优（−36%）** |
| **P99** | **1 286 ms** | **746 ms** | **diff 显著更优（−42%）** |
| 最大延迟 | 1 606 ms | 782 ms | diff 更优 |
| >1 s 请求数 | 3（3%） | 0（0%） | diff 更优 |

### 核心结论
1. **same 模式存在冷启动链式长尾问题**：测试初期多个新建 pod 同时被首次命中，连续触发 >1 s 尖刺，对 P95/P99 影响显著。
2. **diff 模式的尾延迟稳定收敛**：即使有首次命中，最大延迟控制在 800 ms 内，**无任何 >1 s 请求**。
3. **稳态性能两者相当**，差异主要来自尾部。

### 建议
- **生产环境若关心 SLA / 长尾（P95/P99）→ 选择 diff 模式**。
- 如果业务可接受偶发冷启动尖刺，且希望最大化利用单 runtime 资源 → 可继续用 same，但需要：
  - 部署后预热（warm-up 请求）；
  - 或在网关侧开启慢启动/重试以屏蔽尾部尖刺。
